## Successes

1. **End-to-end MVP works** — Navionics export → Bosco import flow functions in real conditions. Seb demoed to sailor friends who are eager for more. This is the strongest product validation signal at this stage.

2. **Story spec quality** — "Previous Story Intelligence" pattern in each story eliminated context loss between stories. Every story contained code examples, anti-patterns, and explicit references to existing patterns. Dev agent had complete implementation context.

3. **Architecture integrity** — 3-tier Supabase containment respected across all 11 stories. ESLint `no-restricted-imports` enforcement worked. ActionResponse `{ data, error }` pattern adopted universally — no Server Action throws.

4. **Test discipline** — 0 → 177 tests with zero regressions. Mock Supabase client pattern established in Story 1-2 reused consistently. Each story added tests without breaking existing ones.

5. **Story 1-1 Dev Notes as reference** — Captured critical breaking changes (Next.js 16, Tailwind CSS 4, Zod 4) that served as authoritative reference for the entire project. Prevented repeated mistakes.

6. **Reusable patterns established early** — ActionResponse type, data layer pattern, messages.ts i18n pattern, Server Action validation — all defined in Epic 1 and reused without modification across Epic 2.

---

## Challenges

1. **UI/UX not visually validated** — Gap between design spec and actual rendered output. No UX review step after implementation. Optimized for delivery velocity, not perceived quality. Seb reports display issues on mobile.

2. **Stopover reliability issues** — Merge radius (2km) too small for large ports. Seb's Le Havre example: two points in the same port create duplicate stopovers. Nominatim returns inconsistent place names (streets instead of cities). In-memory cache on Vercel serverless evaporates on cold starts.

3. **Superficial client-side error handling** — ActionResponse pattern is solid server-side, but UI shows generic toast messages. No retry logic, no contextual error messages, no graceful degradation guidance for users.

4. **Story 2-4b (Leg Deletion) not planned** — Discovered mid-sprint as a gap. Sailors need to delete incorrectly imported tracks. Planning phase missed CRUD completeness analysis.

5. **Story 2-1 → 2-3 dependency** — AC-3 (track rendering) blocked by legs table that didn't exist yet. Story remained artificially "in-progress". Dependency not detected during planning.

6. **Zero E2E tests** — Playwright configured since Story 1-1 but `tests/e2e/` still empty after 11 stories. All confidence rests on unit tests + manual testing.

7. **Inconsistent story statuses** — Some stories marked "review" in their files but "done" in sprint-status.yaml. No clear process to synchronize.

8. **Dialog @base-ui/react API confusion** — `render` prop vs `asChild` (Radix pattern) tripped up development in Stories 1-4 and 2-6. Documented each time but not captured in CLAUDE.md.

9. **Supabase local unavailability** — Story 1-4 had to manually add types because local Supabase wasn't running. Risk of type drift vs actual schema.

---

## Key Insights

1. **Velocity has a UX cost** — 11 stories in ~3 days without visual review creates experience debt. The product works but doesn't feel polished.

2. **Unit tests don't catch experience problems** — 177 passing tests didn't detect the Le Havre duplicate stopovers or display issues. Real-world data testing is essential.

3. **Synthetic test data lies** — GPX fixtures with 2-3 points don't represent Navionics exports. Real GPS data is chaotic (drift, port-scale movements, signal gaps).

4. **First user validation is priceless** — Seb's demo to sailor friends provided more actionable feedback than all automated checks combined.
