## Successes

1. **Execution velocity** — 3 stories delivered in essentially 2 working days (March 18 primary development, March 22 polish). Epic 3 was the fastest epic yet despite introducing complex features (route animation, OG images, SSR public pages).

2. **High-priority technical debt resolved** — Both critical items from the Epic 1-2 retro were addressed *before* Epic 3 started:
   - Stopover deduplication (HIGH): Name-based dedup with 25nm distance guard + merge radius increased to 5km. Commit `40b55e8`.
   - Persistent geocode cache (MEDIUM): Two-level cache (L1 in-memory + L2 Supabase DB) replacing the volatile in-memory-only cache. Migration `20260317160000_geocode_cache.sql`.

3. **No surprise stories** — Unlike Epic 2 (which spawned unplanned story 2-4b for leg deletion), Epic 3 completed exactly as scoped. The CRUD gap analysis concern from the retro was structurally avoided — Epic 3 is entirely read-only public pages.

4. **Architecture integrity maintained** — All `@supabase` imports confined to Tier 1. Server/Client component boundaries correct. All map components properly use `dynamic({ ssr: false })`. Public pages use Server Components for SSR with Client Components only for interactive elements.

5. **SSR + SEO done right** — Public voyage page (`/[username]/[slug]`) renders complete HTML server-side with JSON-LD structured data, dynamic Open Graph images, and proper metadata. Public profile page follows the same pattern. This is a strong foundation for discoverability.

6. **Test growth continues** — 51 new tests added across data layer, components, utilities, and accessibility. Specific highlight: accessibility tests for both map and stopover markers (`mapAccessibility.test.ts`, `stopoverMarkerAccessibility.test.ts`).

7. **Client-side geocoding architecture** — Story 3.2 moved geocoding to client-side during import (via `/api/geocode` proxy), eliminating the Vercel serverless timeout issue that plagued Epic 2. Pre-resolved names are passed through to `persistStopovers`. This is a clean architectural improvement.

   | Action Item | Status | Notes |
   |-------------|--------|-------|

   | Real Navionics GPX fixtures | NOT DONE | Zero `.gpx` files in repo; `.gitignore` excludes them |
   | CRUD completeness analysis | NOT DONE | Structurally unnecessary for Epic 3 (read-only), but process not established |

2. **Story status desynchronization worsened** — The problem from Epic 1-2 propagated. Current state:
   - Story 3-1: `review` in file, `done` in YAML
   - Story 3-2: `review` in file, `done` in YAML
   - Story 3-3: `done` in both (consistent)
   - 7/11 Epic 1-2 stories still have stale statuses

3. **Fire-and-forget recurrence** — Despite the Epic 1-2 retro explicitly identifying fire-and-forget as a critical Vercel bug, a new instance exists at `src/app/voyage/[id]/stopover/actions.ts:297`:
   ```typescript
   void reverseGeocodeServer(parsed.data.latitude, parsed.data.longitude).then(...)
   ```
   This is in `repositionStopover` — when a user moves a stopover, the re-geocode will silently fail on Vercel production. **This is a regression of the exact same pattern that was fixed in the retro remediation.**

4. **Zero E2E tests** — `tests/e2e/` directory remains empty after 14 stories across 3 epics. Playwright is configured since Story 1-1 but has never been used. The team agreement "E2E tests cover at minimum the happy path of each epic" has been broken three times.

5. **i18n gaps in OG images** — ~15 hardcoded English strings in `opengraph-image.tsx` files (both profile and voyage). Labels like `"Distance"`, `"Days"`, `"Ports"`, `"Countries"`, `"Shared from Bosco"` are not in `messages.ts`. The page components properly externalize strings, but the OG image generators were overlooked.

6. **Test coverage gaps for Epic 3 features** — No tests for:
   - Public profile page (`/[username]/page.tsx`)
   - Public voyage page (`/[username]/[slug]/page.tsx`)
   - `PublicVoyageContent.tsx` orchestration
   - OG image generation (both files)
   - `StatsBar.tsx` and `BoatBadge.tsx` components
   - `VoyageSettingsForm.tsx` / settings actions

7. **Commit messages lack detail** — Epic 3 commits are predominantly `"3.X wip"` or `"3.X wip"`. Compare with the pre-Epic-3 fix commits which had descriptive messages explaining *what* and *why*. This makes git history archaeology harder.

## Key Insights

1. **Process improvements need enforcement, not just agreement** — All 4 action items were agreed upon but none were followed. The pattern suggests that action items written in a retro document are not sufficient — they need to be embedded in the workflow (e.g., story template checklist, pre-commit validation, automation).

2. **Technical debt fixes stick, process changes don't** — Both high-priority technical fixes (stopover dedup, geocode cache) were implemented successfully. These were concrete code changes with clear implementation paths. Process changes (visual QA, status sync, GPX fixtures) require ongoing discipline and had no enforcement mechanism.

3. **Fire-and-forget is a systemic pattern, not a one-time bug** — Its recurrence in `repositionStopover` after being explicitly fixed for `persistStopovers` suggests the pattern is intuitive for the developer (start async work, don't wait). A linting rule or code review checkpoint is needed.

4. **OG images are a testing blind spot** — They generate images server-side, making them hard to test with Vitest. But they contain business logic (stat formatting, fallback text, layout decisions) that should be validated.

5. **Velocity remains excellent** — 3 epics in ~7 days with 228 passing tests, zero type errors, zero lint issues, and clean architecture. The project is on track for MVP completion with Epic 4.

### Team Agreements (Revised)

Previous agreements were all broken. Revised approach — fewer, more enforceable:
